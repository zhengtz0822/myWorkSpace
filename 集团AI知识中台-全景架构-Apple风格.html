01

为什么要做

知识散落在各系统，AI 能力各自为战。不是技术问题，是架构问题。

痛点一
知识不可查
同一个工艺参数，MES 里的和 Wiki 里的对不上，找不到权威来源。工程师查一个问题要翻 3 个系统、5 份文档。
痛点二
知识不可联
设备故障 → 维修方案 → 备件库存，这些知识散落在不同系统，无法串联推理，全靠老师傅经验。
痛点三
AI 能力碎片化
每个系统单独接大模型，Prompt 和工具无法复用，重复造轮子。模型 API 分散采购，成本不可控。
痛点四
效率短板
工程师查一个工艺问题耗时 30 分钟起步。异常分析报告人工汇总 2 小时。AI 投入分散，ROI 难以衡量。

技术选型决策

工具接入协议 MCP
Function Calling 的上层标准化，解决工具发现/注册/鉴权/跨模型复用
备选：原生 Function Calling（无标准化，不可复用）
知识检索方案 GraphRAG
结合图结构推理 + 向量检索，比纯向量 RAG 召回率和推理深度更好
备选：纯向量 RAG（无法做关联推理）
Agent 互联协议 A2A
Google 推出的 Agent-to-Agent 标准协议，解决多 Agent 协作
备选：自定义协议（维护成本高）
Agent 运行时 Harness 模式
沙箱隔离 + 反馈闭环 + 重试降级，生产级可靠性
备选：裸 Agent（不可控）
图数据库 / 向量数据库 / Embedding 待评估
需评估：查询性能、社区生态、国产化要求、运维成本。方案与存储引擎解耦，可替换
Neo4j / NebulaGraph / TuGraph · Milvus / Weaviate / Qdrant · bge-m3 / text-embedding-v3
03

核心能力

不只是对话式问答，而是执行式 AI。能查、能联、能用、能推理。

🔍
MES 知识查询
自然语言查工艺参数、设备状态、生产记录。GraphRAG 关联推理，不是关键词匹配。
🔧
MCP 工具调用
标准化协议操作业务系统——查询、修改、提交。工具即插即用，跨模型复用。
🧠
知识图谱推理
故障根因分析、工艺参数关联、异常溯源。图结构推理，比纯文本检索深一层。
🤝
Agent 协作
多 Agent 分工完成复杂任务：查数据 → 分析 → 生成报告。A2A 标准协议。
📊
智能报表
对话式生成生产报告、异常分析、趋势预测。自然语言 → 结构化数据 → 图表。
📚
知识管理
知识录入、更新、审核、版本管理。知识生命周期管理，不是存了就不管。

06

风险管控

不回避风险，正面应对。每个风险都有预案。

模型 API 不稳定 中概率 · 高影响
多模型路由 + 降级策略 + 本地小模型兜底
知识质量不达标 高概率 · 中影响
人工审核 + 自动校验 + Bad Case 持续优化
业务方配合度低 中概率 · 高影响
从 MES 单场景切入，快速出成果建立信心
图数据库选型失误 低概率 · 中影响
GraphRAG 方案与存储引擎解耦，可替换
工期延期 中概率 · 中影响
渐进式建设，每期有独立交付，不依赖后续
