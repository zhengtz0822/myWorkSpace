01

为什么要做

🔍

知识不可查

同一工艺参数，MES 里的和 Wiki 里的对不上，找不到权威来源

🔗

知识不可联

设备故障→维修方案→备件库存，散落不同系统，无法串联推理

🤖

知识不可用

每个系统单独接大模型，Prompt 和工具无法复用，重复造轮子

⏱️

效率短板

工程师查一个工艺问题要翻 3 个系统、5 份文档，耗时 30 分钟起步

💰

成本短板

各系统独立采购模型 API，无统一调度，成本不可控

知识中台的本质：不是再建一个系统，而是建一层知识即服务的基础设施
让知识从"存着等人翻" → "主动推送、按需检索、关联推理"

03

技术选型决策

MCP 是 Function Calling 的上层标准化协议，不是替代关系
路径：Function Calling（模型原生能力）→ MCP（标准化协议）→ A2A（Agent 互联）

🔌 工具接入协议

MCP

Function Calling 的上层标准化，解决工具发现/注册/鉴权/跨模型复用

备选：原生 Function Calling（无标准化，不可复用）

🧠 知识检索方案

GraphRAG

结合图结构推理 + 向量检索，比纯向量 RAG 召回率和推理深度更好

备选：纯向量 RAG（无法做关联推理）

🤝 Agent 互联协议

A2A

Google 推出的 Agent-to-Agent 标准协议，解决多 Agent 协作

备选：自定义协议（维护成本高）

⚙️ Agent 运行时

Harness 模式

沙箱隔离 + 反馈闭环 + 重试降级，生产级可靠性

备选：裸 Agent（不可控）

🗄️ 图数据库

待定

需评估：查询性能、社区生态、国产化要求、运维成本

候选：Neo4j / NebulaGraph / TuGraph

📐 向量数据库

待定

需评估：性能、易用性、私有化部署

候选：Milvus / Weaviate / Qdrant

知识建模层次

📋 事实层

设备参数、工艺指标、操作记录
例："A 线温度上限 85°C"

📐 规则层

工艺约束、安全规则、操作规程
例："温度超 85°C 触发告警"

💡 经验层

故障案例、处理方案、专家经验
例："A 线温度异常通常是冷却水阀故障"

🧠 推理层

因果关系、关联推理
例："冷却水阀故障 → 温度异常 → 产品缺陷"

05

核心能力

🔍

知识工程能力

  • 知识抽取：从文档/数据库/日志自动抽取实体和关系
  • 知识融合：不同来源同一实体做合并、消歧
  • 知识推理：基于图结构做因果推理、关联发现
  • 知识更新：增量更新，新知识自动关联已有知识
  • 知识评测：质量评分、覆盖率统计、过期检测
⚙️

Agent 工程能力

  • 沙箱执行：Docker 隔离，限制 CPU/内存/网络
  • 反馈闭环：失败 → 错误喂回 → 修正 → 重试
  • 重试/降级：最多 N 次重试，超时降级到简化方案
  • 预算控制：单次任务 token 费用上限，超预算停止
  • 评测体系：Golden Dataset 回归 + 线上反馈收集
🛡️

安全 & 权限保障

  • 输入安全：Prompt 注入防护、工具参数校验
  • 执行安全：沙箱隔离、高危操作人工审批
  • 数据安全：知识按业务域隔离、敏感信息脱敏
  • 权限控制：按角色控制工具调用权限、知识访问范围
  • 审计日志：所有 Agent 操作留痕可追溯

价值收益

📈 业务价值

工艺参数查询从 30 分钟降到秒级；设备故障排查从靠老师傅到知识民主化；异常报告从 2 小时人工汇总到 5 分钟自动生成

🔗 数据价值

跨系统知识可查、可联、可用；统一口径，同一知识只有一个权威来源；从"查数据"到"给建议"，知识驱动决策

🏗️ 技术价值

AI 接入从每个系统单独接到中台统一服务；MCP 工具即插即用，跨模型复用；架构从烟囱式到分层可扩展

📋 管理价值

Prompt 版本管理、工具权限统一、知识质量可评测；沙箱执行、高危审批、审计日志；数据隔离、敏感信息脱敏
